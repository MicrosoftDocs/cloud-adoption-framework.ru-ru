# <a name="democratize-data"></a>Данные более демократичным

Угля, топливо и человеческий потенциал были тремя самыми ценными активами на протяжении всей промышленной революции. Эти активы созданы компании, сдвинутые рынки и в конечном итоге изменили Rise of Nations. В цифровой экономике есть три одинаковых важных ресурса: данные, устройства и потенциальные возможности. В каждом из этих ресурсов есть великолепные возможности для инноваций. В рамках любых инноваций данные представляют собой новый объем.

В каждой компании уже есть карманных устройств данных, которые можно использовать для быстрого поиска и удовлетворения потребностей клиентов. К сожалению, процесс интеллектуального анализа данных, позволяющих повысить производительность инноваций, занимает много времени. Многие из наиболее ценных решений для нужд клиентов неудовлетворенных, так как нужные пользователи не могут получить доступ к нужным им данным.

Демократизатион данных — это процесс получения этих данных в нужные руки для внедрения инноваций. Этот процесс может принимать несколько фигур, но обычно включает решения для полученных или интегрированных необработанных данных, централизованное разделение данных, совместное использование данных и защиту данных. В случае успеха эксперты компании могут использовать данные для тестирования этих данных. Во многих случаях группы по внедрению в облако могут [создавать сопереживание с](./build.md) использованием только данных, что позволяет быстро влиять на существующие потребности клиентов.

## <a name="process-of-democratizing-data"></a>Процесс обработки данных демократизинг

На следующих этапах будут описаны решения и подходы, необходимые для внедрения решения, более демократичным данные. Для создания конкретного решения не требуется каждый этап. Однако каждый из них должен оцениваться при создании решения для гипотезы клиента. Каждый предоставляет уникальный подход к созданию инновационных решений.

![Процесс для данных демократизинг](../../_images/innovate/democratize-data.png)

### <a name="share-data"></a>Совместное использование данных

При [создании с помощью Customer сопереживание](./build.md)все процессы сосредоточены на заказчике в соответствии с техническим решением. Данные демократизинг не являются исключением, поэтому мы начнем с предоставления общего доступа к данным. Чтобы более демократичным данные, он должен включать решение, которое совместно использует данные с потребителем данных. Потребитель данных может быть прямым клиентом или прокси-сервером, который принимает решения для клиентов. Утвержденные потребители данных могут анализировать, опрашивать и сообщать о централизованных данных без поддержки ИТ-специалистов.

Многие успешные нововведения были запущены как специалисты MVP, которые обеспечивают ручные, управляемые данными процессы от имени клиента. В этой модели Concierge сотрудник является потребителем данных. Этот сотрудник использует данные для помощи клиентам. Каждый раз, когда клиент применяет поддержку вручную, можно проверить и проверить гипотезу. Такой подход часто является экономичным средством тестирования гипотезы, ориентированной на клиентов, до того, как она сильно полагается на интегрированные решения.

Основные средства для совместного использования данных непосредственно с потребителями данных, включают в себя самостоятельную отчетность или данные, внедренные в другие интерфейсы, с помощью таких средств, как [Power BI](https://docs.microsoft.com/power-bi).

> [!NOTE]
> Прежде чем предоставлять общий доступ к данным, важно учитывать следующие разделы. Для совместного использования данных может потребоваться управление, чтобы обеспечить защиту общих данных. Дальнейшее распределение данных между несколькими облаками и может потребовать централизованного использования. Большая часть данных может даже находиться в приложениях, для которых потребуется сбор данных перед предоставлением общего доступа.

### <a name="govern-data"></a>Управление данными

Совместное использование данных может быстро создать MVP, который можно использовать в беседах с клиентами. Однако только данные — это просто данные. Чтобы превратить эти общие данные в набор знаний, требующих вмешательства, часто требуется немного больше. После того как гипотеза проверена через общий доступ к данным, на следующем этапе разработки часто осуществляется управление данными.

Управление данными — это обширная тема, которая может потребовать собственной специализированной платформы. Это выходит за пределы области [инфраструктуры внедрения в облако](../../index.md). Однако существует несколько аспектов системы управления данными, которые следует учитывать, как только будет проверена гипотеза клиента. Ниже приведено несколько примеров этих вопросов.

- **Учитывается ли общий доступ к данным?** [Данные должны классифицироваться](../../govern/policy-compliance/data-classification.md) перед любым общедоступным общим доступом для защиты интересов клиентов и компании.
- **Если данные конфиденциальны, они защищены?** Защита конфиденциальных данных должна быть обязательной для любых демократичными данных. В примере рабочей нагрузки, на которой нацелены [решения по защите данных](https://docs.microsoft.com/azure/architecture/data-guide/scenarios/securing-data-solutions.md) , предоставляется несколько ссылок для защиты данных.
- **Каталог данных?** Сведения о сборе данных для совместного использования будут способствовать долгосрочному управлению данными. Средства для документирования данных, например каталога данных Azure, могут значительно упростить этот процесс в облаке. Рекомендации по [заметкам данных](https://docs.microsoft.com/azure/data-catalog/data-catalog-how-to-annotate) и [документации по источникам данных](https://docs.microsoft.com/azure/data-catalog/data-catalog-how-to-documentation) помогут ускорить процесс.

Если демократизатион данных важна для клиентов, Управление общими данными должно быть где-то в плане выпуска для защиты клиентов, потребителей данных и компании.

### <a name="centralize-data"></a>Централизация данных

При нарушении работы данных в ИТ-среде возможности внедрения инноваций могут быть чрезвычайно ограниченными, дорогостоящими и длительными. Облако предоставляет новые возможности для централизации данных в различных приемниках данных. Если требуется выполнить централизованную работу с несколькими источниками данных для [сборки с помощью Customer сопереживание](./build.md) , облако может ускорить тестирование этих целей.

> [!CAUTION]
> Централизованное хранение данных — это распространенная точка риска в любом процессе внедрения инноваций. Если централизованная работа с данными является [техническим скачком](./build.md#reduce-complexity-and-delay-technical-spikes), а не источником значения клиента, то рекомендуется отложить централизованное хранение до проверки гипотезы клиента.

Если требуется централизованное хранение данных, сначала необходимо определить соответствующее хранилище данных для централизованных данных. Рекомендуемый подход заключается в том, чтобы создать хранилище данных в облаке. Этот масштабируемый параметр обеспечит центральное расположение всех данных. Этот тип решения доступен в режиме интерактивной аналитической обработки (OLAP) или в параметрах больших данных.

Эталонные архитектуры для решений [OLAP](https://docs.microsoft.com/azure/architecture/data-guide/relational-data/online-analytical-processing) и [больших данных](https://docs.microsoft.com/azure/architecture/data-guide/big-data) могут помочь в выборе наиболее подходящего решения в Azure. Если требуется гибридное решение, Эталонная архитектура для [расширения локальных данных](https://docs.microsoft.com/azure/architecture/data-guide/scenarios/hybrid-on-premises-and-cloud) также может помочь ускорить разработку решений.

> [!IMPORTANT]
> В зависимости от потребностей клиента и согласованного решения может быть достаточно более простое решение. Архитектор облачных решений должен выработать с более низкими затратами, что может привести к более быстрой проверке гипотезы клиента, особенно при ранних разработках. В разделе Сбор данных ниже приводится несколько сценариев, которые могут запрашивать другое решение.

### <a name="collect-data"></a>Сбор данных

Если данные должны быть централизованными для доставки потребности клиентов, очень вероятно, что данные также потребуется собирать из различных источников и перемещать в централизованное хранилище данных. Существует два основных вида сбора данных: интеграция и прием. Каждый из них предоставляет различные средства сбора данных.

**Интеграция:** Существующие данные, находящиеся в существующем хранилище данных, можно интегрировать в централизованное хранилище данных с помощью традиционных методов перемещения данных. Это особенно распространено для сценариев, в которых используется многооблачное хранилище данных. Эти методы состоят из извлечения данных из существующего хранилища данных. Затем загружая данные в центральное хранилище данных. В какой-то момент в этом процессе данные часто преобразуются, чтобы быть более пригодными и актуальными в центральном хранилище.

Облачные инструменты применяют эти методики к средствам оплаты за использование, сокращая барьер на попадание в сбор данных и централизованную детализацию. Такие средства, как служба переноса данных и фабрика данных, являются двумя распространенными примерами в Azure. Эталонная архитектура [фабрики данных с хранилищем данных OLAP](https://docs.microsoft.com/azure/architecture/data-guide/relational-data/etl) предоставляет пример такого решения.

Прием **:** Некоторые данные не находятся в существующем хранилище данных. Если эти временные данные являются основным источником инноваций, следует учитывать альтернативные подходы. Временные данные можно найти в различных существующих источниках, таких как приложения, API, потоки данных, устройства IoT, блокчейн, кэш приложения, мультимедийное содержимое или даже неструктурированные файлы.

Эти различные формы данных можно интегрировать в центральное хранилище данных в решении OLAP или больших данных. Однако для ранних итераций циклов "сборка-мера-обучение" в оперативном решении транзакционной обработки транзакций (OLTP) может быть больше, чем достаточно для проверки гипотезы клиента. Решения OLTP не являются самым высоким качеством для любого сценария создания отчетов. Однако при [создании с помощью Customer сопереживание](./build.md) сосредоточиться на потребности клиентов, они имеют приоритет над техническими решениями. После проверки гипотезы клиента может потребоваться более подходящая платформа. Эталонная архитектура [хранилищ данных OLTP](https://docs.microsoft.com/azure/architecture/data-guide/relational-data/online-transaction-processing) помогает определить, какое хранилище данных наиболее подходит для вашего решения.

**Виртуализация:** Интеграция и прием данных иногда могут замедлять инновации. Если решение для виртуализации данных уже доступно, это может быть более релевантным подходом. Прием и интеграция могут дублировать требования к хранению и разработке, добавлять задержку данных, увеличивать уязвимую контактную зону, внедрять проблемы с качеством и увеличивать усилия по управлению. Виртуализация данных — это более современная альтернатива, которая оставляет исходные данные в одном месте и создает сквозные или кэшированные запросы к исходным данным.

SQL Server 2017 и хранилище данных SQL Azure поддерживают [polybase](/sql/relational-databases/polybase/polybase-guide) , что является подходом к виртуализации данных, наиболее часто используемым в Azure.
