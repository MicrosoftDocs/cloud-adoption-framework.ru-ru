# <a name="engage-through-applications"></a>Участие в приложениях

Как обсуждалось в статье о [демократизинг данных](./data.md), данные являются новыми. В этом заключается большинство нововведений в сфере цифровой экономике. Основываясь на этом аналоге, приложения являются станцией и инфраструктурой, необходимой для получения топлива в нужных руках.

В некоторых случаях данные достаточно для изменения и удовлетворения потребностей клиентов. Чаще всего для решения клиентов требуются приложения для формирования данных и создания интерфейса. Приложения — это способ привлечения пользователя. Они являются домашним для процессов, необходимых для реагирования на триггеры клиентов. Это клиенты, способные предоставлять данные и получать рекомендации. В этой статье приводятся некоторые принципы, которые будут использоваться для согласования правильного решения приложения на основе подтверждений, которые должны быть проверены.

## <a name="shared-code"></a>Общий код

Группа разработчиков, которая может быстрее и точно реагировать на Отзывы клиентов, изменения рынка и возможности внедрения инноваций, приведет к появлению новых рынков в инновациях. Первый принцип инновационных приложений описан в [обзоре принципов роста](./learn.md#growth-mindset), "поделиться кодом". Инновации со временем могут поступать только из фокуса культуры. Чтобы поддерживать инновации, требуются различные перспективы и вклады.

Чтобы подготовиться к инновациям, вся разработка приложений должна начаться с общего репозитория кода. Наиболее распространенным средством для управления репозиториями кода является [GitHub](https://guides.github.com/), что позволяет создавать общий репозиторий кода несколькими щелчками. Кроме того, можно использовать функцию [Azure Repos](/azure/devops/repos/get-started/what-is-repos?view=azure-devops) Azure DevOps для создания репозитория [Git](/azure/devops/repos/get-started/what-is-repos?view=azure-devops#git) или [Team Foundation](/azure/devops/repos/get-started/what-is-repos?view=azure-devops#tfvc) .

## <a name="citizen-developers"></a>Разработчики в отношении граждан

Профессиональные разработчики являются важнейшим компонентом инноваций. Когда гипотеза доказывается точно в масштабе, профессиональным разработчикам необходимо стабилизировать и подготовить решение для масштабирования. Большинство принципов, упоминаемых в этой статье, потребует поддержки профессиональных разработчиков. К сожалению, текущие тенденции предполагают, что у профессиональных разработчиков есть больше вакансий, а у них есть разработчики. Кроме того, стоимость и темп инноваций могут быть менее приемлемыми, когда требуются специально настроенные возможности профессиональной разработки. Разработчики, предоставляющие собственные возможности, позволяют масштабировать усилия по разработке и ускорить тестирование на ранней гипотезе.

Разработчики-создатели могут быть в разумных подходах при проверке ранних подтверждений с помощью таких средств, как [PowerApps](https://docs.microsoft.com/powerapps/powerapps-overview) , для интерфейсов приложений, [искусственного интеллекта](/powerapps/use-ai-builder) для процессов и прогнозов, [Microsoft Flow](https://docs.microsoft.com/flow) для рабочих процессов или [Power BI](https://docs.microsoft.com/power-bi) данных. затрат.

> [!NOTE]
> При работе с разработчиками-авторами для тестирования этих данных рекомендуется, чтобы профессиональный разработчик предоставил поддержку, проверку и рекомендации. После проверки гипотезы процесс перехода приложения в более надежную модель программирования ускоряет работу на новшествах. Включение профессиональных разработчиков в ранние определения процессов может привести к более поздним переходам.

## <a name="modern-web-experiences"></a>Современные веб-интерфейсы

Если приложение или опыт требуется для удовлетворения потребностей клиентов, современные веб-приложения могут быть самым быстрым способом удовлетворить эти потребности. Современные веб-возможности могут быстро привлекать внутренних или внешних клиентов и обеспечить быструю итерацию решения.

Служба приложений Azure предоставляет среду размещения для приложений, которая устраняет нагрузку по управлению инфраструктурой и исправлению ОС. Она обеспечивает автоматизацию масштабирования в соответствии с потребностями пользователей, но ограничивается определенными ограничениями для сохранения затрат. Служба приложений предоставляет поддержку первого класса для таких языков, как ASP.NET, ASP.NET Core, Java, Ruby, Node. js, PHP или Python. Если у вас есть необходимость размещения другого стека времени выполнения, веб-приложение для контейнера предоставляет возможность быстрого и простого размещения контейнера DOCKER в среде службы приложений, позволяя размещать пользовательский стек кода в среде, получающей вас из бусин сервера. дистанцион.

## <a name="cloud-native-solutions"></a>Собственные решения в облаке

Облачные собственные приложения создаются с нуля. Облачные собственные приложения оптимизированы для масштабирования и производительности облака. Облачные приложения обычно строятся с использованием микрослужб, бессерверных, основанных на событиях или основанных на контейнерах подходов. Чаще всего облачные решения используют сочетание архитектур микрослужб, управляемых служб и непрерывной поставки для достижения максимальной надежности и ускорения рынка.

Облачное решение позволяет централизованным группам разработчиков сохранять контроль над бизнес-логикой без необходимости использовать монолитные централизованные решения. Этот тип решения также создает привязку, обеспечивающую согласованность при работе с другими разработчиками и современные возможности. Наконец, облачные решения предоставляют инновационные ускорители, освобождая программистов и профессиональных разработчиков для безопасного создания инноваций и с минимальными блокировками.

## <a name="innovate-through-existing-solutions"></a>Инновации в существующих решениях

Многие клиентские данные могут быть предоставлены современной версией существующего решения. Когда текущая бизнес-логика соответствует потребностям клиента (или в самом деле близко), вы можете ускорить инновации, создав на основе современного решения.

Большинство форм модернизации, включая небольшой рефакторинг приложения, включены в [методологию миграции](../../migrate/index.md) в облачной инфраструктуре внедрения. Эта методология помогает группам облачного внедрения использовать процессы для переноса [цифрового пространства](../../digital-estate/index.md) в облако. В рамках [руководств по миграции Azure](../../migrate/azure-migration-guide/index.md) предусмотрен оптимальный подход к той же методологии, который подходит для небольшого количества рабочих нагрузок или даже одного приложения.

После переноса и современного решения существует множество способов, которые можно использовать при создании новых инновационных решений для нужд клиентов. Например, [разработчики](#citizen-developers) , создающие собственные, могут протестировать данные, а профессиональные разработчики могли создавать [современные веб-интерфейсы](#modern-web-experiences) или [облачные решения](#cloud-native-solutions).

### <a name="extend-an-existing-solution"></a>Расширение существующего решения

Расширение решения — одна из распространенных форм модернизации. Этот подход может быть самым быстрым путем к инновациям, если на гипотезе клиента справедливы следующие условия:

- Существующая бизнес-логика должна соответствовать существующим потребностям клиента (или близко к встрече).
- Улучшенная работа лучше соответствует потребностям конкретного клиента когорту.
- Бизнес-логика, необходимая для решения MVP, была централизованной, обычно через [N-уровневое](/azure/architecture/guide/architecture-styles/n-tier), веб-службы, API или конструкцию [микрослужб](/azure/architecture/guide/architecture-styles/microservices) . Этот подход состоит из упаковки существующего решения с новым интерфейсом, размещенным в облаке. В Azure это решение, скорее всего, будет активно в службах приложений Azure.

### <a name="rebuild-an-existing-solution"></a>Перестроение существующего решения

Если приложение не может быть легко расширено, может потребоваться выполнить рефакторинг решения. При таком подходе Рабочая нагрузка переносится в облако. После переноса части приложения изменяются или дублируются, как веб-службы или [микрослужбы](/azure/architecture/guide/architecture-styles/microservices), которые развертываются параллельно с существующим решением. Параллельное решение на основе служб может рассматриваться как расширенное решение. Это решение будет просто переносить существующее решение с новым интерфейсом, размещенным в облаке. В Azure это решение, скорее всего, будет активно в службах приложений Azure.

> [!CAUTION]
> Рефакторинг и реструктуризация решений, а также централизованная бизнес-логика могут быстро стать значительным [временем, в](./build.md#reduce-complexity-and-delay-technical-spikes)отличие от источника ценности клиента. Это является риском для инноваций, особенно на раннем этапе проверки гипотезы. С небольшой творческой целью в проектировании решения должен быть путь к MVP, который не требует переработки существующих решений. Рекомендуется отложить рефакторинг, пока начальная гипотеза не сможет быть проверена в масштабе.

## <a name="operating-model-innovations"></a>Нововведения в рабочей модели

Помимо современных нововведений, связанных с созданием приложений, существуют нововведения, связанные с работой приложений. Подходы породили много перемещений Организации. Одним из наиболее выразительного является перемещение в [облако по отношению к облачному центру](../../organize/cloud-center-of-excellence.md) наиболее эффективно работающих моделей. При полном штате и готовности бизнес-группы могут предоставлять собственную операционную поддержку для решения.

Тип модели оперативного управления самообслуживания, который находится в облачном центре, позволяет более тесным элементам управления и быстрее выполнять итерации в среде решения. Это достигается за счет передачи операционному управлению и отчетности бизнес-группе.

Если цель заключается в масштабировании или выполнении глобального спроса на существующее решение, этот подход может быть достаточным для проверки гипотезы клиента. После переноса и незначительного современного бизнеса Бизнес-Группа могла бы масштабировать решения, чтобы протестировать различные данные о когортые, которые связаны с производительностью, глобальным распределением или другими потребностями клиентов, которые ему мешают. Operations.

## <a name="reduce-overhead-and-management"></a>Сокращение издержек и управление

Чем больше необходимо поддерживать в решении, тем медленнее это решение будет выполнять итерацию. Ускорьте инновации, уменьшая влияние операций на доступную скорость.

Чтобы подготовиться к множеству итераций, необходимых для предоставления инновационного решения, важно заранее подумать. Сокращайте эксплуатационные нагрузки на ранних этапах процесса, отдавая предпочтение бессерверным параметрам.

В Azure параметры бессерверных приложений могут включать [службу приложений Azure](https://docs.microsoft.com/azure/app-service/overview), [Service Fabric](https://docs.microsoft.com/azure/architecture/example-scenario/infrastructure/service-fabric-microservices), [контейнеры](https://docs.microsoft.com/azure/architecture/cloud-adoption/migrate/azure-best-practices/contoso-migration-rearchitect-container-sql)и т. д.

В параллельном режиме Azure обеспечивает бессерверные параметры данных транзакций, что также сокращает затраты. [Список продуктов базы данных](https://docs.microsoft.com/azure/#pivot=products&panel=databases) содержит параметры для размещения данных без необходимости полной платформы данных.
