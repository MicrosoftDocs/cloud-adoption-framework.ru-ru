# <a name="ambient-experiences-interact-with-devices"></a>Внешний интерфейс: взаимодействие с устройствами

В процессе [сборки с помощью сопереживание](./build.md)мы обсуждали три теста, поступающих от клиентов, и постарайтесь, чтобы клиент продолжался, масштабироваться на основе клиента когорты. Каждый тест гипотезы потребует усилий и итераций на подходе к внедрению. В этой статье приводятся подробные сведения о некоторых усовершенствованных подходах к снижению усилий благодаря **внешнему интерфейсу**. При взаимодействии с устройствами, а не с приложением, клиент может, скорее всего, сначала превратиться в решение.

## <a name="ambient-experiences"></a>Внешний интерфейс

Окружающая среда — это цифровой опыт, который связан с непосредственным окружающим окружением. Решение, которое применяет внешние функции, позволяющие удовлетворить потребности клиента. По возможности решение отвечает требованиям клиентов, не прибегая к естественному потоку действий.

Жизнь цифровой экономичности полностью превышена. Мы все бомбардед с помощью социальных сетей, электронной почты, веб-, визуальных и текстовом сообщений, каждый из которых является риском раскрытия. Риск вычитания увеличивается с каждой секунды между точкой клиента и взаимодействием с решением. Бесчисленные клиенты теряются в течение этого короткого промежутка времени. Чтобы повысить эффективность повторного перехода, сократите число отсчетов, уменьшив время до решения (TTS).

## <a name="interacting-with-devices"></a>Взаимодействие с устройствами

Стандартный веб-интерфейс — это наиболее распространенный метод разработки приложений, используемый для удовлетворения потребностей клиента. Предположение, помогут в этом подходе, заключается в том, что клиент будет находиться перед компьютером. Если клиент согласованно соответствует своим потребностям, а перед переносным компьютером, создайте веб-приложение. Это веб-приложение будет предоставлять «окружающую среду» для этого клиента в этом сценарии. Но мы понимаем, что этот сценарий все еще менее вероятен в современных обществах.

![Внешний интерфейс](../../_images/innovate/ambient-experiences.png)

Внешний интерфейс, как показано выше, обычно требует больше веб-приложения. Благодаря [измерению](./measure.md) и [изучению клиента](./learn.md) поведение, которое ведет к триггеру потребности клиента, можно наблюдать, относить и использовать для создания более внешнего интерфейса. Ниже приведено несколько подходов к интеграции внешних решений в свои данные. Дополнительные сведения о каждом из них приведены в следующих параграфах.

- **[Возможности для мобильных устройств](#mobile-experience)** . как и ноутбук, мобильные приложения обычно являются частью окружающей среды клиентов. В некоторых сценариях это может обеспечить достаточный уровень интерактивности для обеспечения внешнего взаимодействия решения.
- **[Смешанная реальность](#mixed-reality):** Иногда для создания внешнего взаимодействия необходимо изменить естественную окружающую подстановку клиента. Это создает несколько ложных дел, в которых клиент может взаимодействовать с решением и иметь потребность в его выполнении. В этом случае решение является окружающим в результате ложной реальности.
- **[Интегрированная реальность](#integrated-reality):** Поместивсь ближе к реальному окружению, встроенные решения на практике посвящены использованию устройства, которое существует в реальности клиента, для интеграции решения в естественное поведение. Виртуальный помощник — это отличный пример интеграции реальность в окружающую среду. Менее известный вариант — использовать технологии "Интернет вещей" (IoT) для интеграции устройств, которые уже существуют в режиме окружающей среды клиента.
- **[Скорректированная реальность](#adjusted-reality):** Когда какое-либо из описанных выше внешних решений использует прогнозный анализ в облаке для определения и предоставления взаимодействия с клиентом с помощью естественной окружающей среды, решение имеет скорректированную реальность.

Понимание потребности клиентов и измерение воздействия на клиентов поможет определить, требуется ли взаимодействие устройства или окружающая среда для проверки гипотезы. В каждой из этих точек данных можно найти лучшее решение в следующих разделах.

## <a name="mobile-experience"></a>Возможности мобильных устройств

Первый этап окружающей среды заключается в том, чтобы приступить к работе с компьютера. Современные потребители и бизнес-специалисты переходят между мобильными устройствами и ПК. Каждая из платформ или устройств, используемых вашим клиентом, создает новый потенциальный опыт для клиента. Добавление мобильного интерфейса, расширяющего основное решение, является самым быстрым способом улучшения интеграции в немедленную окружающую среду клиента. Хотя мобильное устройство далеко от внешнего, оно может быть ближе к точке потребности клиента.

Когда клиенты находятся на мобильном устройстве и часто изменяют расположения, это может быть наиболее важной формой внешнего интерфейса для конкретного решения. Одним из распространенных источников инноваций за последнее десятилетие было расширение существующих решений для интеграции возможностей мобильных устройств.

Примеры такого подхода можно просмотреть в службе приложений Azure. Во время ранних итераций [функция веб-приложения службы приложений Azure](/azure/app-service/overview) может использоваться для проверки гипотезы. По мере того, как все они становятся более сложными, [функция мобильного приложения служб приложений Azure](/azure/app-service-mobile/) может расширить веб-приложение для работы на различных мобильных платформах.

## <a name="mixed-reality"></a>Смешанная реальность

Следующий уровень зрелости для внешних интерфейсов — это приращение или репликация окружающей среды клиента с помощью решений смешанной реальности. При таком подходе решение станет более внешним, так как оно создает расширение реальность для работы клиента в.

> [!IMPORTANT]
> Если требуется устройство виртуальной реальности (VR), *которое еще не является частью непосредственных окружающих или естественных поведений*, то расширение или виртуальная реальность является более предпочтительным вариантом и меньше окружающих возможностей.

Такой вид интерфейса все чаще используется для удаленных сотрудников. Использование этих возможностей растет еще быстрее в отраслях, требующих совместной работы или специальных навыков, которые не доступны на локальном рынке. Один из распространенных сценариев, в которых в качестве части естественного поведения используется Расширенная модель, является централизованной поддержкой реализации сложного продукта для удаленной силы. В этих сценариях Центральная группа поддержки и удаленный сотрудник могут использовать дополненную реальность для устранения неполадок или установки продукта.

В описанном выше сценарии или в других похожих сценариях примером внешнего интерфейса будет использование пространственных привязок. Пространственные привязки позволяют создавать смешанные возможности работы с объектами, сохраняющими их расположение на устройствах с течением времени. С помощью пространственных привязок можно записывать, записывать и сохранять определенные действия, обеспечивая внешний интерфейс, при следующем запуске пользователя в этой расширенной среде. [Пространственные привязки Azure](https://docs.microsoft.com/azure/spatial-anchors/overview) — это служба, которая перемещает эту логику в облако, что позволяет совместно использовать возможности на разных устройствах и даже в решениях.

## <a name="integrated-reality"></a>Интегрированная реальность

Помимо мобильной или даже смешанной реальности, используется интегрированная реальность. В этом подходе цель состоит в том, чтобы полностью удалить цифровой интерфейс. Все, что нас в мире, — это устройства с возможностями вычислений и подключения. Эти устройства можно использовать для получения данных из непосредственных окружающих сред без необходимости накасаться телефона, ноутбука или устройства VR.

Такая форма работы идеально подходит, когда в одной и той же области окружающей среды возникает одна разновидность устройства. Распространенные сценарии: заводские этаже, лифты или даже ваш автомобиль. Эти типы больших устройств уже содержат мощность вычислений. Вы также можете использовать данные с самого устройства для определения поведения клиентов и отправки этих поведений в облако. Автоматическое отслеживание данных о поведении клиентов значительно сокращает потребность в вводе данных клиентом. Кроме того, в качестве цикла обратной связи можно использовать возможности работы с Интернетом, мобильными устройствами или VR, чтобы поделиться информацией, полученной от решения интегрированной реальности.

Примеры интегрированной реальности в Azure могут включать:

- [Решения azure "Интернет вещей" (IOT)](https://docs.microsoft.com/azure/iot-fundamentals)— это коллекция служб в Azure, которая помогает управлять устройствами и потоками данных от этих устройств в облаке и обратно для конечных пользователей.
- [Azure Sphere](/azure-sphere), сочетание оборудования и программного обеспечения Azure Sphere является небезопасным способом обеспечения безопасной передачи данных между устройством и решениями Azure IOT.
- [Azure Kinect Developers Kit](https://docs.microsoft.com/azure/Kinect-dk), ии датчики AI с улучшенными концепциями компьютерных концепций и речевыми моделями, которые могут получать визуальные и аудиоданные-данные из непосредственных окружающих элементов и передавать их в свое решение.

Все три можно использовать для получения данных в естественном окружающем блоке, в то время, когда клиент нуждается в. После этого решение может ответить на эти входные данные, чтобы решить эту проблему, иногда перед тем, как клиент даже знает, что был вызван триггер.

## <a name="adjusted-reality"></a>Скорректированная реальность

Самая высокая форма внешнего интерфейса — это настраивается реальность, которая часто называется внешней логикой. Скорректированная реальность — это подход к использованию информации из решения для изменения реальности клиента без необходимости непосредственно взаимодействовать с приложением. При таком подходе приложение, которое вы изначально создали для подтверждения гипотезы, может быть больше не релевантным. Вместо этого устройства в среде упрощают входные и выходные данные, чтобы удовлетворить потребности клиентов.

Виртуальные помощники и интеллектуальные динамики могут предоставить отличный пример скорректированной реальности. И один из них — это пример простой интегрированной реальности. Добавьте к решению для смарт-динамиков смарт-освещение и сенсор движения, и вы можете легко создать базовое решение с включением огней при вводе комнаты.

Заводские этаже по всему миру предоставляют дополнительные сценарии скорректированной реальности. На ранних этапах интегрированной реальности датчики на устройствах обнаружили условия, например перегревается, и сообщила о необходимости изменения сотрудника через приложение. В скорректированной реальности клиент по-прежнему может участвовать. Но цикл обратной связи является более строгим. В производственном этаже с скорректированной назначением одно устройство может обнаружить перегрев на жизненном станке в пределах строки сборки. В другом месте второе устройство будет немного замедлять работу, чтобы позволить компьютеру выдать охлаждение и возобновить скорость при разрешении условия. В этом сценарии клиент является второй рукой. Клиент использует приложение для установки правил и понимания того, как эти правила затронули рабочую среду, но не обязательно должны быть частью цикла обратной связи.

Службы Azure в предыдущем разделе, [решения azure "Интернет вещей" (IOT)](https://docs.microsoft.com/azure/iot-fundamentals), [Azure Sphere](/azure-sphere)и [Azure Kinect Developers Kit](https://docs.microsoft.com/azure/Kinect-dk) могут быть компонентами решения скорректированной реальности. Исходное приложение и бизнес-логика будут служить посредником между входными данными окружающей среды и изменениями, которые необходимо внести в физическую среду.

Еще один пример скорректированной реальности — это создание цифрового двойникаа, который представляет собой цифровое представление физического устройства, представленное через компьютерный, мобильный или смешанный формат. В отличие от менее сложных трехмерных моделей, цифровой двойника отражает данные, собранные с фактического устройства в физической среде. Это позволяет пользователю взаимодействовать с цифровым представлением способами, которые никогда не могут быть выполнены в реальном мире. При таком подходе физические устройства настраиваются в среде смешанной реальности. Но решение по-прежнему собирает данные из интегрированного решения реальность и использует его для формирования факта текущей окружающей среды клиента.

В Azure цифровые двойников создаются и доступны через службу, именуемую [Azure Digital двойников](https://docs.microsoft.com/azure/digital-twins/about-digital-twins).
