# <a name="application-migration-patterns-and-examples"></a>Шаблоны переноса приложений и примеры

В этом разделе положений Cloud Adoption Framework приведены примеры нескольких распространенных сценариев переноса, демонстрирующие, как можно перенести локальную инфраструктуру в облако [Microsoft Azure](https://azure.microsoft.com/overview/what-is-azure).

## <a name="introduction"></a>Вступление

Azure предоставляет доступ к комплексному набору облачных служб. С помощью этих служб вы, как разработчики и ИТ-специалисты, можете создавать, развертывать и администрировать приложения с помощью разных средств и инфраструктур в глобальной сети центров обработки данных. Если ваша компания столкнется с проблемами, связанными с переходом на цифровые технологии, облако Azure поможет вам оптимизировать ресурсы и операции, наладить взаимодействие с клиентами и сотрудниками и трансформировать свои продукты.

Однако мы признаем, что даже при всех преимуществах облака Azure, таких как скорость и гибкость, минимальные затраты, высокая производительность и надежность, у многих организаций в какой-то момент возникнет потребность в запуске локальных центров обработки данных. В ответ на препятствия, возникающие в процессе внедрения облачных технологий, Azure предоставляет стратегию гибридного облака, которая создает мосты между локальными центрами обработки данных и общедоступным облаком Azure. Это, например, использование таких облачных ресурсов Azure, как служба Azure Backup, для защиты локальных ресурсов или получение информации о локальных рабочих нагрузках с помощью служб аналитики Azure.

В рамках стратегии гибридного облака Azure предоставляет постоянно расширяющиеся решения для переноса локальных приложений и рабочих нагрузок в облако. С помощью простых шагов вы можете в полном объеме оценить локальные ресурсы, чтобы понять, как они будут работать в облаке Azure. Затем, с учетом этой оценки, можно без опасений переносить ресурсы в Azure. Когда ресурсы будут доступны и запущены в Azure, вы сможете оптимизировать их для сохранения и улучшения доступа, повышения гибкости, безопасности и надежности.

## <a name="migration-patterns"></a>Шаблоны миграции

Стратегий миграции в облако четыре: повторное размещения, рефакторинг, перепроектирование и перестроение. Используемая стратегия зависит от бизнес-факторов и целей миграции. Вы можете реализовать несколько шаблонов. Например, можно выбрать повторное размещение простых приложений или приложений, которые не являются критически важными для вашего бизнеса, но изменить архитектуру приложений, которые являются более сложными и критически важными для бизнеса. Рассмотрим эти шаблоны.

<!-- markdownlint-disable MD033 -->

**Шаблон** | **Определение** | **Время использования**
--- | --- | ---
**Повторное размещение** | Часто называется миграцией с _прогнозированием и сдвигом_ . Этот вариант не требует изменений кода, позволяя быстро перенести существующие приложения в Azure. Каждое приложение переносится в облачную среду как есть, без риска и затрат, связанных с изменениями кода. | При необходимости быстро переместить приложения в облако. <br><br> При необходимости переместить приложение без его изменения. <br><br> Если приложения спроектированы таким образом, чтобы они могли воспользоваться преимуществами масштабируемости [Azure IaaS](https://azure.microsoft.com/overview/what-is-iaas) после миграции. <br><br> Если приложения важны для бизнеса, но немедленные изменения в их возможностях не требуются.
**Рефакторинг** | Как правило, для репакетирования требуется внести минимальные изменения в приложения, чтобы они могли подключаться к [Azure PaaS](https://azure.microsoft.com/overview/what-is-paas)и использовать облачные предложения. <br><br> Например, можно перенести существующие приложения в Службу приложений Azure или Службу Azure Kubernetes (AKS). <br><br> Кроме того, реляционные и нереляционные базы данных можно перерабатывать в такие параметры, как Управляемый экземпляр Базы данных SQL Azure, база данных Azure для MySQL, база данных Azure для PostgreSQL и Azure Cosmos DB. | Если ваше приложение можно легко перепаковать для работы в Azure. <br><br> Если требуется применить инновационные рекомендации DevOps, предоставляемые Azure, или же вы рассматриваете реализацию DevOps с использованием стратегии контейнеров для рабочих нагрузок. <br><br> Перед применением рефакторинга необходимо оценить переносимость существующей базы кода и имеющиеся навыки разработки.
**Перепроектирование** | Перепроектирование для миграции предусматривает изменение и расширение функциональности приложения и базы кода, чтобы оптимизировать архитектуру приложения для обеспечения масштабируемости в облаке. <br><br> Например, можно разбить монолитное приложение на группу микрослужб, которые работают совместно и легко масштабируются. <br><br> Или же можно перепроектировать реляционные и нереляционные базы данных в полностью управляемое решение базы данных, например в Управляемый экземпляр Базы данных SQL Azure, Базу данных Azure для MySQL, Базу данных Azure для PostgreSQL и Azure Cosmos DB. | Если приложениям требуется масштабная переработка для внедрения новых возможностей или для более эффективной работы на облачной платформе. <br><br> Если вы хотите использовать существующие вложения приложений, соблюдайте требования к масштабируемости, примените инновационные методики DevOps и сократите использование виртуальных машин.
**Перестроение** | Перестроение предусматривает еще один шаг — сборку приложения заново с использованием облачных технологий Azure. <br><br> Например, вы можете создавать приложения с зеленым полем с помощью [собственных облачных](https://azure.microsoft.com/overview/cloudnative) технологий, таких как функции Azure, Azure AI, Управляемый экземпляр Базы данных SQL Azure и Azure Cosmos DB. | Если требуется быстрая разработка, а существующие приложения обладают ограниченной функциональностью и временем существования. <br><br> Если вы готовы ускорить бизнес-инновации (включая рекомендации по DevOps от Azure), построить новые приложения с помощью облачных технологий и воспользоваться преимуществами прогресса ИИ, блокчейна и Интернета вещей.

### <a name="demo-apps"></a>Демонстрационные приложения

В примерах статей, приведенных в этом разделе, используются два демонстрационных приложения: SmartHotel360 и Остиккет.

- **SmartHotel360:** Это приложение было разработано корпорацией Майкрософт в качестве тестового приложения, которое можно использовать при работе с Azure. Оно предоставляется с открытым исходным кодом и доступно для скачивания с сайта [GitHub](https://github.com/Microsoft/SmartHotel360). Это приложение ASP.NET, подключенное к базе данных SQL Server. В сценариях, рассматриваемых в этих статьях, текущая версия данного приложения развертывается на двух виртуальных машинах VMware под управлением Windows Server 2008 R2 и SQL Server 2008 R2. Эти виртуальные машины приложения размещаются в локальной среде и управляются сервером vCenter Server.
- **остиккет:** Приложение для работы с билетами службы поддержки с открытым кодом, которое работает в Linux. Скачать эту версию можно с [GitHub](https://github.com/osTicket/osTicket). В сценариях, обсуждаемых в этих статьях, текущая версия этого приложения развертывается локально на двух виртуальных машинах VMware под управлением Ubuntu 16,04 LTS с использованием Apache 2, PHP 7,0 и MySQL 5,7.
