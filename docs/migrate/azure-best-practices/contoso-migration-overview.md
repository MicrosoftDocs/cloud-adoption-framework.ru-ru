# <a name="overview-of-application-migration-examples-for-azure"></a>Обзор примеров переноса приложений для Azure

В этом разделе платформы внедрения облаков приведены примеры некоторых распространенных сценариев миграции и показано, как можно перенести локальную инфраструктуру в [Microsoft Azure](https://azure.microsoft.com/overview/what-is-azure).

## <a name="introduction"></a>Введение

Azure предоставляет доступ к комплексному набору облачных служб. Разработчики и ИТ-специалисты могут использовать эти службы для создания, развертывания приложений и управления ими на различных средствах и платформах через глобальную сеть центров обработки данных. Если ваша компания столкнется с проблемами, связанными с переходом на цифровые технологии, облако Azure поможет вам оптимизировать ресурсы и операции, наладить взаимодействие с клиентами и сотрудниками и трансформировать свои продукты.

Azure распознает, что даже со всеми преимуществами, предоставляемыми облаком, с точки зрения скорости и гибкости, минимальными затратами, производительностью и надежностью, многим организациям приходится запускать локальные центры обработки данных в течение некоторого времени. В ответ на барьеры внедрения в облако Azure предоставляет стратегию гибридного облака, которая создает мосты между локальными центрами обработки данных и общедоступным облаком Azure (например, с помощью облачных ресурсов Azure, таких как Azure Backup, для защиты локальных ресурсов или Azure Analytics, чтобы получить ценные сведения о локальных рабочих нагрузках).

В рамках стратегии гибридного облака Azure предоставляет растущие решения для переноса локальных приложений и рабочих нагрузок в облако. С помощью простых шагов вы можете в полном объеме оценить локальные ресурсы, чтобы понять, как они будут работать в облаке Azure. Затем, с учетом этой оценки, можно без опасений переносить ресурсы в Azure. Когда ресурсы будут доступны и запущены в Azure, вы сможете оптимизировать их для сохранения и улучшения доступа, повышения гибкости, безопасности и надежности.

## <a name="migration-patterns"></a>Шаблоны миграции

Стратегий миграции в облако четыре: повторное размещения, рефакторинг, перепроектирование и перестроение. Используемая стратегия зависит от бизнес-факторов и целей миграции. Вы можете реализовать несколько шаблонов. Например, можно выбрать повторное размещение некритических приложений при реконструировании приложений, которые являются более сложными и критически важными для бизнеса. Рассмотрим следующие шаблоны:

| Шаблон | Определение | Назначение |
| --- | --- | --- |
| **Повторное размещение** | Часто называется миграцией с прогнозированием и сдвигом. Этот параметр не требует изменения кода и позволяет быстро перенести существующие приложения в Azure. Каждое приложение переносится как есть, чтобы пользоваться преимуществами облака без риска и затрат, связанных с изменениями кода. | Когда необходимо быстро переместить приложения в облако. <br><br> Если вы хотите переместить приложение, не изменяя его. <br><br> Если приложения спроектированы таким образом, чтобы они могли воспользоваться преимуществами масштабируемости [Azure IaaS](https://azure.microsoft.com/overview/what-is-iaas) после миграции. <br><br> Когда приложения важны для вашего бизнеса, но вам не нужно немедленно изменять возможности приложения. |
| **Рефакторинг** | Как правило, для репакетирования требуется внести минимальные изменения в приложения, чтобы они могли подключаться к [Azure PaaS](https://azure.microsoft.com/overview/what-is-paas) и использовать облачные предложения. <br><br> Например, можно перенести существующие приложения в службу приложений Azure или службу Kubernetes Azure (AKS). <br><br> Также можно выполнить рефакторинг реляционных и нереляционных баз данных в такие параметры, как Управляемый экземпляр Azure SQL, база данных Azure для MySQL, база данных Azure для PostgreSQL и Azure Cosmos DB. | Если приложение можно легко переупаковать для работы в Azure. <br><br> Если вы хотите применить инновационные методики DevOps, предоставляемые Azure, или вы думаете о DevOps с помощью стратегии контейнеров для рабочих нагрузок. <br><br> Перед применением рефакторинга необходимо оценить переносимость существующей базы кода и имеющиеся навыки разработки. |
| **Перепроектирование** | Изменение архитектуры для миграции сосредоточено на изменении и расширении функциональных возможностей приложения и базы кода для оптимизации архитектуры приложения для масштабируемости в облаке. <br><br> Например, можно разбить монолитное приложение на группу микрослужб, которые работают совместно и легко масштабируются. <br><br> Вы также можете изменять архитектуру реляционных и нереляционных баз данных до полностью управляемого решения базы данных, например Azure SQL Управляемый экземпляр, базы данных Azure для MySQL, базы данных Azure для PostgreSQL и Azure Cosmos DB. | Когда вашим приложениям требуются основные редакции для внедрения новых возможностей или эффективной работы на облачной платформе. <br><br> Если вы хотите использовать существующие вложения приложений, соблюдайте требования к масштабируемости, примените инновационные методики DevOps и сократите использование виртуальных машин. |
| **Перестроение** | Перестроение выполняет шаг вперед, перестраивая приложение с нуля с помощью облачных технологий Azure. <br><br> Например, можно создавать приложения нуля с [собственными](https://azure.microsoft.com/overview/cloudnative) технологиями, такими как функции Azure, AI, управляемый экземпляр SQL и Azure Cosmos DB. | Если требуется быстрая разработка, а существующие приложения имеют ограниченную функциональность и время существования. <br><br> Если вы готовы ускорить бизнес-инновации (включая рекомендации по DevOps от Azure), построить новые приложения с помощью облачных технологий и воспользоваться преимуществами прогресса ИИ, блокчейна и Интернета вещей. |

В примерах статей, приведенных в этом разделе, используются два демонстрационных приложения: SmartHotel360 и Остиккет.

**SmartHotel360:** Это приложение было разработано корпорацией Майкрософт в качестве тестового приложения, которое можно использовать при работе с Azure. Оно предоставляется с открытым исходным кодом и доступно для скачивания с сайта [GitHub](https://github.com/Microsoft/SmartHotel360). Это приложение ASP.NET, подключенное к базе данных SQL Server. В сценариях, обсуждаемых в этих статьях, текущая версия этого приложения развертывается на двух виртуальных машинах VMware под управлением Windows Server 2008 R2 и SQL Server 2008 R2. Эти виртуальные машины приложений размещаются локально и управляются vCenter Server.

**остиккет:** Приложение для работы с билетами службы поддержки с открытым кодом, которое работает в Linux. Скачать эту версию можно с [GitHub](https://github.com/osTicket/osTicket). В сценариях, обсуждаемых в этих статьях, текущая версия этого приложения развертывается локально на двух виртуальных машинах VMware под управлением Ubuntu 16,04 LTS с использованием Apache 2, PHP 7,0 и MySQL 5,7.
