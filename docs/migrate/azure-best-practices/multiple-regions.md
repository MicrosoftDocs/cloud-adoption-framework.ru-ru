# <a name="azure-regions-decision-guide"></a>Рекомендации по решениям для регионов Azure

Azure включает в себя множество регионов по всему миру. Каждый [регион Azure](https://azure.microsoft.com/global-infrastructure/geographies/) имеет определенные характеристики, из-за которых выбор самого региона становится невероятно важным. К ним относятся доступные службы, емкость, ограничения и независимости:

- **Доступные службы:** Службы, развернутые в каждом регионе, отличаются в зависимости от различных факторов. Выберите регион для рабочей нагрузки, который содержит нужную службу. Дополнительные сведения см. в статье [Доступность продуктов по регионам](https://azure.microsoft.com/global-infrastructure/services/).
- **Емкость:** Для каждого региона предусмотрена максимальная емкость. Это может повлиять на типы подписок, которые могут развертывать типы служб и при каких обстоятельствах. Это отличается от квоты подписки. Если вы планируете миграцию крупномасштабного центра обработки данных в Azure, вам может потребоваться обратиться в локальную группу полей Azure или диспетчер учетных записей, чтобы убедиться, что вы можете выполнить развертывание в нужном масштабе.
- **Ограничения:** Некоторые ограничения размещаются в развернутых службах в определенных регионах. Например, некоторые регионы доступны только в качестве целевого расположения для резервного копирования или отработки отказа. Другие ограничения, которые следует учитывать, — это [требования к обеспечению независимости данных](https://azure.microsoft.com/global-infrastructure/geographies/).
- **Независимости:** Некоторые регионы выделены для конкретных сущностей независимых. Хотя все регионы — регионы Azure, эти регионы независимых полностью изолированы от остальной части Azure. Они не обязательно управляются корпорацией Майкрософт и могут быть ограничены определенными типами клиентов. Независимые регионы перечислены ниже.
  - [Azure China 21Vianet](https://azure.microsoft.com/global-infrastructure/geographies/)
  - [Azure](https://azure.microsoft.com/global-infrastructure/geographies/)для Германии: Azure для Германии является нерекомендуемым в Германии.
  - [Azure для государственных организаций США](https://azure.microsoft.com/global-infrastructure/government/)
  - Два региона в [Австралии](https://azure.microsoft.com/global-infrastructure/geographies/) управляются корпорацией Майкрософт, но предоставляются для государственных организаций, клиентов и подрядчиков. Таким образом, эти регионы содержат клиентские ограничения, аналогичные другим независимых облакам.

## <a name="operate-in-multiple-geographic-regions"></a>Работа в нескольких географических регионах

Если предприятия работают в нескольких географических регионах, что может быть необходимым для обеспечения устойчивости, это порождает потенциальную сложность в следующих формах:

- распределение ресурсов;
- использование профилей пользовательского доступа;
- обеспечение соответствия нормативным требованиям;
- обеспечение региональной устойчивости.

Выбор региона очень важен для общей стратегии внедрения в облако. Для начала рассмотрим вопросы, связанные с сетью.

## <a name="network-considerations"></a>Рекомендации по сети

Для реализации любого надежного облачного развертывания нужно хорошо спланировать сеть, учитывая особенности регионов Azure. Следует учитывать следующее:

- Регионы Azure развертываются парами. В случае сбоя катастрофического региона другой регион в пределах той же геоошибочной границы обозначается как парный регион. Рассмотрите возможность развертывания в парных регионах в качестве основной и дополнительной стратегии устойчивости. Единственным исключением из этой стратегии является `Brazil South` , которое сопряжено с `South Central US` . Дополнительные сведения см. в статье [Непрерывность бизнес-процессов и аварийное восстановление в службах BizTalk: пары регионов Azure](/azure/best-practices-availability-paired-regions).

  - Служба хранилища Azure поддерживает [геоизбыточное хранилище (GRS)](/azure/storage/common/storage-redundancy). Это означает, что три копии данных хранятся в основном регионе, а три дополнительные копии хранятся в парном регионе. Невозможно изменить связывание хранилища для GRS.
  - Эта возможность разнесения данных между парными регионами предусмотрена для служб, использующих GRS. Но для этого в приложениях и сети нужно реализовать соответствующую поддержку.
  - Если вы не планируете использовать GRS для поддержки региональных требований к устойчивости, не следует использовать парный регион в качестве получателя. Региональный сбой ощутимо затронет ресурсы в парном регионе при миграции. Вы можете избежать этой ситуации, восстановив на альтернативном сайте и получив дополнительную скорость при восстановлении.
  > [!WARNING]
  > Не используйте Azure GRS для резервного копирования или восстановления виртуальных машин. Вместо этого используйте [Azure Backup](https://azure.microsoft.com/services/backup/) и [Azure Site Recovery](https://azure.microsoft.com/services/site-recovery/), а также [управляемые диски Azure](/azure/virtual-machines/managed-disks-overview)для обеспечения устойчивости рабочей нагрузки "инфраструктура как услуга" (IaaS).

- Azure Backup и Azure Site Recovery поддерживают архитектуру клиентских сетей, обеспечивая региональную устойчивость для решений IaaS и удовлетворяя потребности в резервном копировании данных. Убедитесь, что сеть оптимизирована, поэтому передача данных остается в магистрали Майкрософт и использует [пиринг виртуальных сетей](/azure/virtual-network/virtual-network-peering-overview), если это возможно. Некоторые крупные организации с глобальными развертываниями могут использовать [ExpressRoute Premium](/azure/expressroute/expressroute-introduction), чтобы маршрутизировать трафик между регионами и, возможно, сэкономить на региональных исходящих платежах.

- Группы ресурсов Azure зависят от региона. Однако это нормально для ресурсов в группе ресурсов, охватывающих несколько регионов. Учтите, что в случае региональной ошибки операции плоскости управления для группы ресурсов завершатся сбоем в затронутом регионе, даже если ресурсы в других регионах (внутри этой группы ресурсов) будут продолжать работать. Эти процессы могут повлиять как на структуру сети, так и на структуру группы ресурсов.

- Многие службы "платформа как услуга" (PaaS) в [конечных точках службы](/azure/virtual-network/virtual-network-service-endpoints-overview) поддержки Azure или в [частной связи Azure](/azure/private-link/private-link-overview). Оба эти решения влияют на вашу сеть в зависимости от региональных устойчивости, миграции и управления.

- Многие службы PaaS обеспечивают региональную устойчивость с помощью собственных решений. Например, база данных SQL Azure и Azure Cosmos DB позволяют легко выполнять репликацию в другие регионы. Такие службы, как Azure DNS, не имеют региональных зависимостей. При рассмотрении того, какие службы будут использоваться в процессе внедрения, необходимо четко понимать возможности отработки отказа и шаги восстановления, которые могут потребоваться для каждой службы Azure.

- Помимо развертывания в нескольких регионах для поддержки аварийного восстановления многие организации выбирают развертывание в шаблоне "активный — активный", чтобы не полагаться на отработку отказа. Этот метод обеспечивает дополнительные преимущества глобальной балансировки нагрузки, повышенной отказоустойчивости и увеличения производительности сети. Чтобы воспользоваться преимуществами этого шаблона, ваши приложения должны поддерживать выполнение Active-Active в нескольких регионах.

> [!WARNING]
> Регионы Azure — это высокодоступные конструкции с соглашениями об уровне обслуживания, применяемыми к службам, выполняемым в них. Но не следует использовать зависимость одного региона с критически важными приложениями. Всегда Планируйте региональные сбои и шаги по восстановлению и устранению рисков.

После рассмотрения топологии сети необходимо ознакомиться с дополнительной документацией и выравниванием процессов, которые могут потребоваться. Следующий подход поможет оценить потенциальные проблемы и определить общее направление действий.

- Рассмотрите более надежную реализацию средств обеспечения готовности и системы управления.
- Выполните инвентаризацию затронутых географий. Составьте список затрагиваемых регионов и стран.
- Задокументируйте требования к независимости данных. Действуют ли в указанных странах требования, регулирующие независимость данных?
- Задокументируйте базу пользователей. Повлияет ли миграция облака на сотрудников, партнеров или клиентов в указанной стране?
- Задокументируйте центры обработки данных и ресурсы. Размещены ли в указанной стране ресурсы, которые могут участвовать в процессе миграции?
- Задокументируйте требования к региональной доступности SKU и отработке отказа.

Сопоставьте изменения в процессе миграции, чтобы учесть первоначальные данные инвентаризации.

## <a name="relevance-of-data-sovereignty"></a>Релевантность независимости данных

По всему миру правительственные организации начали устанавливать требования к независимости данных, такие как Общий регламент по защите данных (GDPR). Подобные требования соответствия часто требуют локализации в определенном регионе или даже в определенной стране для защиты граждан. В некоторых случаях данные, относящиеся к клиентам, сотрудникам или партнерам, должны храниться на облачной платформе в регионе, в котором находится сам пользователь.

Решение этой задачи было важным фактором, мотивирующим компании, которые работают по всему миру, выполнить миграцию в облако. Для обеспечения требований соответствия некоторые компании решили развернуть дубликаты ИТ-ресурсов в облачных поставщиках в соответствующем регионе. В приведенной выше таблице в качестве примера этого сценария используется Германия. Этот пример включает клиентов, партнеров и сотрудников, но не ассентс в Германии. Эта компания может выбрать развертывание некоторых ресурсов в центре обработки данных в области GDPR, потенциально используя Немецкий центр обработки данных Azure. Поняв, на какие данные распространяется действие GDPR, команда по внедрению в облако сможет выбрать наилучший подход к миграции для текущей ситуации.

### <a name="why-is-the-location-of-end-users-relevant"></a>Почему важно расположение пользователей?

Компании, обслуживающие пользователей в нескольких странах, разработали технические решения для передачи трафика пользователей. В некоторых случаях это подразумевает локализацию ресурсов. В других сценариях компания может выбрать реализацию глобальных решений Глобальной сети (WAN) для решения разнородных пользовательских баз с помощью решений, направленных на сеть. В любом случае на стратегию миграции могут влиять профили использования этих разнородных конечных пользователей.

Так как компания поддерживает сотрудников, партнеров и клиентов в Германии, не имея в нем центров обработки данных, возможно, эта компания реализовала решение с арендой. Этот тип решения направляет трафик в центры обработки данных в других странах. Эта маршрутизация представляет значительный риск для наблюдаемой производительности перенесенных приложений. Внедрение дополнительных прыжков в установленную и настроенную глобальную сеть может привести к видимому ухудшению работы приложений после миграции. Поиск и устранение этих проблем может значительно задержать завершение проекта.

В каждом из следующих процессов руководство по устранению этих сложностей включает в себя предварительные требования, оценку, миграцию и оптимизацию процессов. Чтобы правильно управлять этой сложностью, важно понимать профили пользователей в каждой стране.

### <a name="why-is-the-location-of-datacenters-relevant"></a>Почему важно расположение центров обработки данных?

Расположение существующих центров обработки данных может повлиять на стратегию миграции. Пример.

**Решения по архитектуре:** Целевой регион является одним из первых шагов в проектировании стратегии миграции. На стратегию часто влияет расположение существующих ресурсов. Кроме того, доступность облачных служб и стоимость единицы этих служб могут изменяться в разных регионах. Понимание того, где находятся текущие и будущие активы, влияет на архитектурные решения и может повлиять на бюджетные оценки.

**Зависимости центра обработки данных:** Данные в предыдущей таблице показывают, что зависимости между различными глобальными центрами обработки данных, скорее всего, будут. Эти зависимости не могут быть четко описаны или понятны во многих организациях, которые работают с этим типом шкалы. Подход компании к оценке профилей пользователей помогает понять некоторые из этих зависимостей в Организации. Кроме того, команда должна изучить дополнительные этапы оценки, которые могут снизить риски и сложности, возникающие в зависимости от зависимостей.

## <a name="implement-the-general-approach"></a>Реализация общего подхода

Следующий подход использует модель, управляемую данными, для устранения сложностей глобальной миграции. Если область миграции включает в себя несколько регионов, Группа внедрения в облако должна оценить следующие аспекты готовности:

- Для независимости данных может потребоваться локализация некоторых ресурсов, но многие ресурсы могут не управляться этими ограничениями на соответствие. Такие вещи, как ведение журналов, отчетность, маршрутизация сети, идентификация и другие центральные ИТ службы, могут быть размещены как общие службы в нескольких подписках или даже в нескольких регионах. Команда по внедрению в облако должна оцениваться с помощью модели общей службы для этих служб, как описано в [эталонной архитектуре для топологии HUB и звезды с общими службами](/azure/architecture/reference-architectures/hybrid-networking/#hub-spoke-network-topology).
- При развертывании нескольких экземпляров схожих сред фабрика среды может создать согласованность, улучшить управление и ускорить развертывание. Благодаря [руководству по организации системы управления для сложных предприятий](../../govern/guides/complex/index.md) вы сможете создать среду, которая охватывает несколько регионов.

Когда команда хорошо работает с базовым подходом и готовой к работе, следует рассмотреть несколько необходимых для обработки данных требований:

- **Общее обнаружение:** Заполните таблицу [сложность документирования](#document-complexity) .
- **Выполните анализ профиля пользователя в каждой затронутой стране:** Важно понимать общую маршрутизацию конечных пользователей на ранних этапах процесса миграции. Изменение глобальных арендованных каналов и добавление таких подключений, как ExpressRoute, в облачный центр обработки данных может потребовать месяцев задержек в сети. Выполните эту задачу как можно раньше в процессе миграции.
- **Первоначальная рационализация цифровых площадей:** Всякий раз, когда сложность вводится в стратегию миграции, необходимо выполнить начальное логическое обоснование цифровых площадок. См. Руководство по [обоснованиям цифровых площадок](../../digital-estate/index.md).
  - **Дополнительные требования к цифровому разследу:** Установите политики тегов, чтобы определить рабочую нагрузку, затронутую требованиями независимости данных. Обязательные теги должны появиться на этапе рационализации цифровых активов, а потом их нужно передать в перенесенные ресурсы.
- **Оцените модель концентратора и звезды:** В распределенных системах часто используются общие зависимости. Эти зависимости часто могут быть реализованы с помощью модели "звезда". Хотя такая модель находится вне области действия процесса миграции, она должна быть помечена для рассмотрения в будущих итерациях [готовых процессов](../../ready/index.md).
- **Приоритет невыполненной работы миграции:** Если сетевые изменения необходимы для поддержки рабочего развертывания рабочей нагрузки, которая поддерживает несколько регионов, Группа стратегии облака должна отрегулировать укрупнение изменений в сети и управлять ими. Более высокий уровень поддержки руководителей помогает ускорить изменение, выполнив команду стратегии для изменения приоритета невыполненной работы и гарантировать, что глобальные рабочие нагрузки не блокируются сетевыми изменениями. Таким рабочим нагрузкам следует назначать приоритеты только после завершения сетевых изменений.

Эти условия помогают определить процессы, которые могут устранить эту сложность во время выполнения стратегии миграции.

## <a name="assess-process-changes"></a>Изменения в процессе оценки

При таком подходе к глобальному ресурсу и пользовательской сложности в сценариях миграции следует добавить несколько ключевых действий для оценки кандидатов на миграцию. Эти действия создают данные для уточнения препятствий и результатов для глобальных пользователей и ресурсов.

### <a name="suggested-action-during-the-assess-process"></a>Предлагаемое действие во время процесса оценки

**Оценка зависимостей между центрами обработки данных:** [Средства визуализации зависимостей в службе "миграция Azure](/azure/migrate/concepts-dependency-visualization) " могут помочь выявить зависимости. Лучше всего использовать эти инструменты перед миграцией. При решении задач глобальной сложности она станет необходимым шагом для процесса оценки. Благодаря [группированию зависимостей](/azure/migrate/how-to-create-group-machine-dependencies) визуализация поможет определить IP-адреса и порты всех ресурсов, необходимых для поддержки рабочей нагрузки.

> [!IMPORTANT]
>
> - Эксперт в предметной области, посвященный размещению ресурсов и схемам IP-адресов, требуется для обнаружения ресурсов, которые находятся во вторичном центре обработки данных.
> - Оцените как подчиненные зависимости, так и клиенты в визуализации, чтобы понять двунаправленные зависимости.

**Выявление глобальных последствий для пользователей:** Выходные данные анализа необходимого профиля пользователя должны указывать любую рабочую нагрузку, затронутую глобальными профилями пользователей. Когда кандидат на миграцию находится в списке затронутой рабочей нагрузки, архитектор, подготавливающий миграцию, должен обратиться к экспертам по сети и операциям. Они помогают проверить маршрутизацию сети и ожидания производительности. Как минимум, архитектура должна включать подключение ExpressRoute между ближайшим центром сетевых операций и Azure. [Эталонная архитектура для подключений ExpressRoute](/azure/architecture/reference-architectures/hybrid-networking/expressroute) может помочь в настройке необходимого подключения.

**Проектирование для обеспечения соответствия:** Выходные данные анализа необходимого профиля пользователя должны указывать любую рабочую нагрузку, затронутую требованиями независимости данных. Во время действий архитектуры процесса оценки назначенный архитектор должен ознакомиться с экспертами по соответствию. Они помогают понять требования к миграции и развертыванию в нескольких регионах. Эти требования значительно повлияют на стратегии проектирования. Эталонные архитектуры для многоязыковых [веб-приложений](/azure/architecture/reference-architectures/app-service-web-app/multi-region) и [многоуровневых многоязыковых приложений](/azure/architecture/reference-architectures/n-tier/multi-region-sql-server) могут помочь в проектировании.

> [!WARNING]
> При использовании любой из указанных выше эталонных архитектур может потребоваться исключить определенные элементы данных из процессов репликации, чтобы они соответствовали требованиям независимости данных. Это приведет к добавлению дополнительного шага в процесс перемещения в рабочую среду.

## <a name="migration-process-changes"></a>Изменения процесса миграции

При переносе приложения, которое должно быть развернуто в нескольких регионах, группа по внедрению в облако должна учитывать ряд соображений. Они состоят из структуры хранилища Azure Site Recovery, разработки конфигурации и сервера обработки, проектирования пропускной способности сети и синхронизации данных.

### <a name="suggested-action-during-the-migration-process"></a>Предлагаемое действие в процессе миграции

**Проектирование хранилища Azure Site Recovery:** Azure Site Recovery — это предлагаемое средство для репликации и синхронизации цифровых активов в облаке в Azure. Site Recovery реплицирует данные о ресурсе в хранилище Site Recovery, которое привязано к определенной подписке в определенном регионе и центре обработки данных Azure. При репликации ресурсов во второй регион может также потребоваться второе хранилище Site Recovery.

**Проектирование конфигурации и сервера обработки:** Site Recovery работает с локальным экземпляром конфигурации и сервера обработки, который привязан к отдельному хранилищу Site Recovery. Это означает, что для упрощения репликации может потребоваться установить второй экземпляр этих серверов в исходном центре обработки данных.

**Проектирование пропускной способности сети:** Во время репликации и текущей синхронизации вы перемещаете двоичные данные по сети из исходного центра обработки данных в хранилище Site Recovery в целевом центре обработки данных Azure. Этот процесс потребляет пропускную способность. При дублировании рабочей нагрузки во второй регион удваивается объем потребляемой пропускной способности. Если пропускная способность ограничена или рабочая нагрузка требует передачи большого объема данных конфигурации или отклонения данных, это может повлиять на время, необходимое для завершения миграции. Что более важно, это может повлиять на работу пользователей или приложений, которые по-прежнему зависят от пропускной способности исходного центра обработки данных.

**Синхронизация данных:** Часто увеличение пропускной способности происходит из синхронизации платформы данных. Как определено в эталонных архитектурах [веб-приложений для нескольких регионов](/azure/architecture/reference-architectures/app-service-web-app/multi-region) и [n-уровневых приложений для нескольких регионов](/azure/architecture/reference-architectures/n-tier/multi-region-sql-server), синхронизация данных часто требуется для того, чтобы поддерживать согласованность приложений. Если это требуемое рабочее состояние приложения, может оказаться разумным выполнить синхронизацию между платформой исходных данных и каждой из облачных платформ. Это необходимо сделать перед переносом ресурсов приложения и среднего уровня.

**Аварийное восстановление из Azure в Azure:** Альтернативный вариант может понизить сложность. Если шкала времени и синхронизация данных приближается к поэтапному развертыванию, [Аварийное восстановление из Azure в Azure](/azure/site-recovery/azure-to-azure-architecture) может оказаться приемлемым решением. В этом сценарии выполняется миграция рабочей нагрузки в первый центр обработки данных Azure с помощью одного хранилища Site Recovery, а также проекта конфигурации или сервера обработки. После тестирования рабочей нагрузки можно восстановить ее во втором центре обработки данных Azure из перенесенных ресурсов. Такой подход позволяет сократить влияние на ресурсы в исходном центре обработки данных и использовать преимущества ускоренной передачи данных и высокой пропускной способности подключения между центрами обработки данных Azure.

> [!NOTE]
> Этот подход может увеличить краткосрочные затраты на миграцию за счет дополнительной оплаты за пропускную способность.

## <a name="optimize-and-promote-process-changes"></a>Изменение процесса оптимизации и перемещения в рабочую среду

По мере реализации глобальной сложности во время оптимизации и повышения уровня, в каждом из дополнительных регионов могут потребоваться дублирующиеся усилия. Если одно развертывание является приемлемым, вам может потребоваться дублировать план Бизнес-тестирования и бизнес-изменения.

### <a name="suggested-action-during-the-optimize-and-promote-process"></a>Предлагаемое действие во время процесса оптимизации и перемещения в рабочую среду

**Оптимизация перед тестированием:** Первоначальное тестирование автоматизации позволяет выявление потенциальных возможностей оптимизации, как и при любых усилиях по переносу. В случае глобальных рабочих нагрузок протестируйте рабочую нагрузку в каждом регионе независимо друг от друга. Незначительные изменения конфигурации в сети или целевом центре обработки данных Azure могут повлиять на производительность.

**Планы изменения бизнеса:** Для любого сложного сценария миграции создайте план Бизнес-изменения. Это гарантирует четкий обмен данными, касающихся изменений в бизнес-процессах, пользовательских интерфейсах и времени действий, необходимых для интеграции изменений. В случае глобальной миграции этот план должен включать в себя рекомендации для пользователей в каждой охваченной географии.

**Бизнес-тестирование:** В сочетании с планом изменения бизнеса в каждом регионе может потребоваться бизнес-тестирование. Это обеспечит достаточную производительность и соответствие измененным шаблонам сетевой маршрутизации.

**Рейси продвижения:** Частое продвижение происходит как одно действие, переправляя рабочий трафик на перенесенные рабочие нагрузки. В случае с глобальными усилиями следует доставлять продвижение в авиарейсах (или заранее определенных коллекций пользователей). Это позволяет команде по вопросам облачной стратегии и команде по внедрению в облако выполнять эффективный мониторинг производительности и обеспечивать поддержку пользователей в каждом регионе. Этапы перемещения в рабочую среду часто контролируются на уровне сети. Для этого изменяется маршрутизация для конкретных диапазонов IP-адресов ресурсов исходной рабочей нагрузки, после чего трафик передается в только что перенесенные ресурсы. После переноса указанной коллекции пользователей можно изменить маршрутизацию для следующей группы.

**Оптимизация рейсов:** Одним из преимуществ рейсов в рекламе является то, что он позволяет выполнять более глубокие наблюдения и дополнительную оптимизацию развернутых ресурсов. По истечении непродолжительного периода использования рабочей среды после первого этапа перемещения предлагается дополнительная корректировка перенесенных ресурсов, если это допускается процедурами ИТ-операций.
