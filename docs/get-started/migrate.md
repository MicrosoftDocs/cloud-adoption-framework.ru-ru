# <a name="get-started-accelerate-migration"></a>Начало работы. Ускорение миграции

Правильное выравнивание бизнеса и ИТ-заинтересованных лиц помогает преодолеть препятствий миграции и повысить эффективность миграции. В этой статье приведены рекомендуемые действия для:

- Выравнивание заинтересованных лиц
- Планирование миграции
- Развертывание целевой зоны
- Перенос первых 10 рабочих нагрузок

Он также помогает реализовать правильные процессы управления и контроля.

Используйте это руководством для оптимизации процессов и материалов, необходимых для согласования общих усилий по миграции. В этом руководством используются методологии облачной инфраструктуры внедрения, которые выделены на этом рисунке.

![Начало работы с миграцией в Azure](../_images/get-started/migration-map.png)

Если ваш сценарий миграции нетипичен, вы можете получить персонализированную оценку готовности к переносу вашей организации с помощью [оценки средства стратегического переноса и подготовки (Smart)](/assessments/?id=strategic-migration-assessment). Используйте его для указания рекомендаций, которые лучше удовлетворяют текущим потребностям.

## <a name="get-started"></a>Приступая к работе

Технические усилия и процесс, необходимые для миграции рабочих нагрузок, относительно просты. Очень важно выполнить процесс миграции эффективно. Уровень готовности к миграции еще больше влияет на временные шкалы и успешное завершение общей миграции.

Чтобы ускорить внедрение, необходимо выполнить действия по поддержке группы внедрения в облако во время миграции. В этом разделе описаны эти итеративные задачи, которые помогут клиентам начать с правильного пути к миграции в облако. Для отображения важности вспомогательных шагов миграция указывается как шаг 10 в этой статье. На практике Группа внедрения в облако, скорее всего, начнет свою первую пилотную миграцию с шагами 4 или 5.

## <a name="step-1-align-stakeholders"></a>Шаг 1. Выровняйте заинтересованных лиц

Чтобы избежать распространенных блокирующих миграций, создайте четкую и краткую бизнес-стратегию для миграции. Выравнивание заинтересованных лиц по мотивации и ожидаемым результатам бизнеса решения, принятые группой внедрения в облако.

- [Мотивация](../strategy/motivations.md). Первый этап стратегического выравнивания — получение соглашения о мотивациях, которые применяют усилия по переносу. Начните с понимания и категоризации причин и общих тем от различных заинтересованных лиц в бизнесе и ИТ-отделах.
- [Результаты бизнеса](../strategy/business-outcomes/index.md). После согласования мотиваций можно захватить желаемые результаты бизнеса. Эти сведения предоставляют четкие метрики, которые можно использовать для измерения общего преобразования.

## <a name="step-3-gather-data-and-analyze-assets-and-workloads"></a>Шаг 3. сбор данных и анализ ресурсов и рабочих нагрузок

Используйте обнаружение и оценку, чтобы улучшить техническое выравнивание и создать план действий для исполнения стратегии. На этом этапе проверьте бизнес-вариант, используя данные о текущей среде состояния. Затем выполните количественный анализ и тщательно прокачественная Оценка рабочих нагрузок с наивысшим приоритетом.

- [Инвентаризация существующих систем](../digital-estate/inventory.md): используйте программный подход, управляемый данными, для понимания текущего состояния. Обнаружение и сбор данных для включения всех действий по оценке.
- [Поэтапное рационализация](../digital-estate/rationalize.md#incremental-rationalization). Оптимизируйте усилия по оценке, чтобы сосредоточиться на качественном анализе всех активов, возможно даже для поддержки бизнес-ситуации. Затем добавьте углубленный анализ для первых 10 рабочих нагрузок, подлежат миграции.

**Результатам**

## <a name="step-5-create-a-migration-plan"></a>Шаг 5. Создание плана миграции

План внедрения в облако обеспечивает ускоренный подход к разработке невыполненной работы проекта. Затем можно изменить невыполненную работу, чтобы отразить результаты обнаружения, рационализацию, необходимые навыки и контрактные отработки.

- [План внедрения в облако](../plan/template.md). Определите план внедрения в облако с помощью базового шаблона.
- [Выравнивание рабочей нагрузки](../plan/workloads.md): определение рабочих нагрузок в невыполненной работе.
- [Выравнивание усилий](../plan/assets.md). Выровняйте активы и рабочие нагрузки в невыполненной работе, чтобы четко определить усилия по приоритетным рабочим нагрузкам.
- [Выравнивание людей и времени](../plan/iteration-paths.md): Создание итерации, скорости (время человека) и выпусков для перенесенных рабочих нагрузок.

**Результатам**

- Разверните шаблон невыполненной работы.
- Обновите шаблон, чтобы отразить первые 10 рабочих нагрузок, подлежат миграции.
- Обновите людей и скорость, чтобы оценить время выпуска.
- Риски временной шкалы:
  - Отсутствие опыта работы с Azure DevOps может замедлять процесс развертывания.
  - Сложность и данные, доступные для каждой рабочей нагрузки, также могут повлиять на временные шкалы.

## <a name="step-7-deploy-and-align-a-landing-zone"></a>Шаг 7. Развертывание и согласование целевой зоны

Все перенесенные ресурсы развертываются в целевой зоне. Начальная зона начинается с простого для поддержки небольших рабочих нагрузок, а затем масштабируется для более сложных рабочих нагрузок с течением времени.

- [Выбор целевой зоны](../ready/landing-zone/index.md): Найдите правильный подход к развертыванию целевой зоны на основе шаблона внедрения. Затем разверните эту стандартизированную базу кода.
- [Расширьте целевую зону](../ready/considerations/index.md): любую начальную точку, выявление пробелов в развернутой размещенной зоне и добавление необходимых компонентов для Организации ресурсов, безопасности, управления, соответствия и операций.

**Результатам**

- Разверните первую целевую зону для развертывания первоначальных миграций с низким риском.
- Разрабатывайте план рефакторинга с помощью облачного центра или Центральной ИТ-группы.
- Риски временной шкалы:
  - Управление, операции и требования к безопасности для первых 10 рабочих нагрузок могут замедлять этот процесс.
  - Рефакторинг первой и последующей целевой зон занимает больше времени, но она должна выполняться параллельно с усилиями по миграции.

## <a name="step-8-migrate-your-first-10-workloads"></a>Шаг 8. Перенос первых 10 рабочих нагрузок

Технические усилия, необходимые для переноса первых 10 рабочих нагрузок, относительно просты. Это также итеративный процесс, который повторяется при переносе дополнительных ресурсов. В этом процессе вы оцениваете рабочие нагрузки, развертываете рабочие нагрузки, а затем освобождаете их в рабочей среде.

![Этапы итеративной миграции: Оценка, развертывание, выпуск](../_images/migrate/methodology-effort-only.png)

Средства миграции в облако позволяют перенести все виртуальные машины в центре обработки данных за один проход или итерацию. Чаще всего выполняется перенос меньшего количества рабочих нагрузок во время каждой итерации. Чтобы разбить миграцию на меньшие приращения, необходимо больше планирования, но это снизит технические риски и влияние управления изменениями Организации.

При каждой итерации группа внедрения облачных технологий повышает эффективность миграции рабочих нагрузок. Эти действия помогут техническим специалистам получить свои возможности:

1. Перенесите первую рабочую нагрузку в подходе IaaS (инфраструктура как услуга) с помощью средств, описанных в статье о [миграции Azure](../migrate/azure-migration-guide/index.md).
2. Разверните раздел Параметры инструментария, чтобы использовать миграцию и модернизации с помощью [примеров миграции](../migrate/azure-best-practices/contoso-migration-overview.md).
3. Разрабатывайте свою техническую стратегию с помощью более широкого подхода, описанного в статье рекомендации по [миграции в облако Azure](../migrate/azure-best-practices/index.md).
4. Повышайте согласованность, надежность и производительность с помощью эффективного подхода к миграции, как описано в [улучшениях процесса миграции](../migrate/migration-considerations/index.md).

**Результатам**

Непрерывное улучшение возможности группы внедрения по переносу рабочих нагрузок.

## <a name="step-9-hand-off-production-workloads-to-cloud-governance"></a>Шаг 9. Передача производственных рабочих нагрузок в Cloud управление

Управление — это ключевой фактор для долгосрочного успеха любых усилий по миграции. Важна скорость миграции и бизнес-воздействие. Но скорость без управления может быть опасной. Ваша организация должна принять решения по управлению, которые соответствуют шаблонам внедрения и требованиям к управлению и соответствию.

- [Подход к управлению](../govern/index.md). в этой методологии описывается процесс мышления о корпоративной политике и процессах. Определив свой подход, вы можете создать дисциплины, необходимые для обеспечения управления в рамках деятельности по внедрению облачных решений.
- [Начальная](../govern/guides/complex/prescriptive-guidance.md)система управления: Изучите дисциплины, необходимые для создания минимального допустимого продукта системы управления (MVP), который служит основой для внедрения.
- [Контрольный тест](https://aka.ms/adopt/assess/govern)управления: выявление пропусков в текущем состоянии вашей организации. Получите персонализированный отчет о тестировании и ознакомьтесь с рекомендациями по началу работы.

**Результатам**

- Развертывание первоначальной системы управления.
- Выполните контрольный тест для управления, чтобы спланировать будущие улучшения.
- Риск временной шкалы:
  - Улучшенные политики и реализация управления могут добавить одну до четырех недель на дисциплину.

- [Определение базового плана управления](../manage/index.md)

- Риски временной шкалы:
  - Ознакомьтесь с книгой: Оценка за один час для каждого владельца приложения.
  - Выполните Microsoft Azureную оценку анализа на основе грамотной архитектуры: Оценка за один час для каждого приложения.

## <a name="next-steps"></a>Дальнейшие действия

Облачная инфраструктура внедрения — это решение жизненного цикла, помогающее начать миграцию. Кроме того, она помогает развитым командам поддерживать миграцию. Следующие команды могут использовать эти дальнейшие действия, чтобы продолжить работу со своими возможностями. Эти параллельные процессы не являются линейными и не должны рассматриваться как блокирующие. Вместо этого каждый представляет собой поток параллельных значений, позволяющий повысить общую готовность Организации к облаку.

| Группа | Следующая итерация |
|--|--|
| Группа по внедрению облака | Используйте [модель миграции](../migrate/migration-considerations/index.md) , чтобы узнать о переходе к фабрике миграции, которая обеспечивает эффективные возможности миграции. |
| Группа по облачной стратегии | Итеративно улучшайте [методологию стратегии](../strategy/index.md) и [методологию плана](../plan/index.md) вместе с планом внедрения. Просмотрите эти обзоры и продолжите итерацию по бизнес-и техническим стратегиям. |
| Группа по облачной платформе | Перейдите на [методологию готовности](../ready/index.md) , чтобы продолжить продвижение всей облачной платформы, поддерживающей миграцию, или других действий по внедрению. |
| &nbsp;Группа по управлению облаком &nbsp; | Используйте [управляемую методологию](../govern/index.md) , чтобы продолжить улучшать процессы управления, политики и дисциплины. |
| Группа по эксплуатации облака | Постройте [методологию управления](../manage/index.md) , чтобы предоставить более широкие операции в Azure. |

Если ваш сценарий миграции нетипичен, вы можете получить персонализированную оценку готовности к переносу вашей организации с помощью [оценки средства стратегического переноса и подготовки (Smart)](/assessments/?id=strategic-migration-assessment). Ответы на интересующие вас вопросы помогут определить, какие рекомендации наилучшим образом отвечают текущим потребностям.
