# <a name="cloud-monitoring-guide-monitoring-strategy-for-cloud-deployment-models"></a>Руководство по мониторингу облака: стратегия мониторинга для моделей облачного развертывания

Эта статья содержит рекомендованную стратегию мониторинга для каждой модели облачного развертывания на основе следующих критериев.

- Вам необходимо поддерживать обязательства по Operations Manager или другим корпоративным платформам мониторинга в связи с интеграцией с процессами ИТ-операций, знаниями и опытом, а также некоторыми функциями, которые пока не доступны в Azure Monitor.
- Необходимо отслеживать рабочие нагрузки как локально, так и в общедоступном облаке или только в облаке.
- Стратегия миграции в облако включает модернизации ИТ и переход к нашим облачным службам и решениям мониторинга.
- У вас могут быть критически важные системы, гаппед или физически изолированные, размещенные в частном облаке или на физическом оборудовании, и их необходимо отслеживать.

Наша стратегия включает поддержку инфраструктуры мониторинга (вычислений, хранилища и рабочих нагрузок сервера), приложения (конечных пользователей, исключений и клиентов), а также сетевых ресурсов для предоставления комплексной перспективы мониторинга, ориентированной на службы.

## <a name="azure-cloud-monitoring"></a>Мониторинг облака Azure

Azure Monitor — это служба собственной платформы Azure, которая предоставляет единый источник для мониторинга ресурсов Azure. Он предназначен для облачных решений, построенных на платформе Azure и поддерживающих бизнес-возможности, основанные на рабочих нагрузках виртуальных машин или сложных архитектурах, использующих микрослужбы и другие ресурсы платформы. Он отслеживает все слои стека, начиная с служб клиента, таких как Azure Active Directory доменных служб, а также событий уровня подписки и работоспособности службы Azure. Кроме того, он отслеживает ресурсы инфраструктуры, такие как виртуальные машины, хранилище и сетевые ресурсы, а также на верхнем уровне приложения. Мониторинг каждой из этих зависимостей и сбор правильных сигналов, которые каждый из них может порождать, дает вам наблюдаемые приложения и необходимую инфраструктуру ключей.

## <a name="hybrid-cloud-monitoring"></a>Гибридный мониторинг облака

Для многих организаций облако должно подходить постепенно, где Гибридная Облачная модель является наиболее распространенным первым шагом в пути. Тщательно выбирайте подходящее подмножество приложений и инфраструктуру, чтобы начать миграцию и избежать перерывов в работе предприятия. Тем не менее, поскольку мы предлагаем две платформы мониторинга, поддерживающие эту облачную модель, лица, принимающие решения, принимают решение о том, какой из них лучше подходит для поддержки своих бизнес-целей и рабочих задач ИТ. Мы собираемся рассмотреть несколько факторов, чтобы устранить неопределенность и получить представление о том, какую платформу следует рассмотреть.

Ниже приведены некоторые ключевые технические аспекты, которые следует учитывать:

- Вам необходимо получить данные из ресурсов Azure, которые поддерживают рабочую нагрузку, и пересылать их в существующие локальные или управляемые средства поставщика услуг.

- Вам нужно сохранить текущие инвестиции в System Center Operations Manager и настроить ее для отслеживания ресурсов IaaS и PaaS, работающих в Azure. При необходимости, так как вы наблюдаете за двумя средами с разными характеристиками в зависимости от требований, вы определяете интеграцию с Azure Monitor поддерживает вашу стратегию.

- В рамках стратегии модернизации для стандартизации одного средства, чтобы снизить затраты и сложность, вы фиксируете Azure Monitor для мониторинга ресурсов в Azure и в корпоративной сети.

### <a name="collect-and-stream-monitoring-data-to-third-party-or-on-premises-tools"></a>Получение и потоковая передача данных мониторинга сторонним или локальным средствам

Для сбора метрик и журналов из инфраструктуры Azure и ресурсов платформы необходимо включить журналы диагностики Azure для этих ресурсов. Кроме того, с помощью виртуальных машин Azure вы можете получать метрики и журналы из операционной системы на виртуальной машине, включив расширение система диагностики Azure. Чтобы переслать диагностические данные, порожденные ресурсами Azure, в локальные средства или поставщик управляемых служб, настройте [концентраторы событий](https://docs.microsoft.com/azure/azure-monitor/platform/diagnostic-logs-stream-event-hubs) для потоковой передачи данных.

### <a name="monitor-with-system-center-operations-manager"></a>Мониторинг с помощью System Center Operations Manager

Хотя System Center Operations Manager изначально разрабатывались как локальное решение для наблюдения за приложениями, рабочими нагрузками и инфраструктурой, работающей в ИТ-среде, оно развивается для включения возможностей мониторинга облака и интегрируется с Azure, Office 365 и Amazon Web Services (AWS). Они могут отслеживать различные среды с помощью пакетов управления, разработанных и обновленных для поддержки этих сред.  

Для клиентов, которые внесли значительные инвестиции в Operations Manager для обеспечения всестороннего мониторинга, тесно интегрированного с управление ИТ-услугамиными процессами и инструментами, или клиентами, которые впервые интегрируются с Azure, вы можете понять вопрос:

- Может Operations Manager продолжать предоставлять ценность и делает его разумным?

- Если функции в Operations Manager правильно подходят для нашей ИТ – организации?

- Интеграция Operations Manager с Azure Monitor предоставляет экономичное и комплексное решение для мониторинга, которое требуется?

Если вы уже приложили к Operations Manager, вам не нужно сосредоточиться на планировании миграции, чтобы заменить ее немедленно. С помощью Azure или других поставщиков облачных служб, существующих в качестве расширения своей локальной сети, Operations Manager может отслеживать Гостевые виртуальные машины и ресурсы Azure, как если бы они находились в корпоративной сети. Для этого требуется надежное сетевое подключение между сетью и виртуальной сетью Azure с достаточной пропускной способностью.

Чтобы отслеживать рабочие нагрузки, выполняемые в Azure, вам потребуется:

- [Пакет управления Azure](https://www.microsoft.com/download/details.aspx?id=50013) для сбора метрик производительности, выпущенных службами Azure, такими как веб-роли и Рабочая роль, Application Insights тесты доступности (WebTest), служебную шину и т. д. Пакет управления использует REST API Azure для наблюдения за доступностью и производительностью этих ресурсов. Некоторые типы служб Azure не имеют метрик или предварительно определенных мониторов в пакете управления, но по-прежнему можно отслеживать с помощью связей, определенных в пакете управления Azure для обнаруженных служб.

- [Пакет управления базы данных SQL Azure](https://www.microsoft.com/download/details.aspx?id=38829) для мониторинга доступности и производительности баз данных SQL Azure и серверов базы данных SQL Azure, использующих REST API Azure и запросы T-sql для SQL Server системных представлений.

- Для мониторинга гостевой ОС и рабочих нагрузок, работающих на виртуальной машине, например SQL Server, IIS или Apache Tomcat, необходимо загрузить и импортировать пакет управления, который поддерживает приложение, службу и операционную систему.

Набор знаний определяется в пакете управления, который описывает, как отслеживать отдельные зависимости и компоненты. Оба пакета управления Azure нуждаются в выполнении набора действий по настройке в Azure и Operations Manager, чтобы начать мониторинг этих ресурсов.

На уровне приложений Operations Manager предлагает базовые возможности наблюдения за производительностью приложений для некоторых устаревших версий .NET и Java. Если некоторые приложения в гибридной облачной среде работают в автономном или изолированном от сети режиме, так что они не могут взаимодействовать с общедоступной облачной службой, то Operations Manager наблюдение за производительностью приложений (APM) может быть приемлемым вариантом для некоторые ограниченные сценарии. Для приложений, не работающих на устаревших платформах, размещенных как локально, так и в любом общедоступном облаке, который разрешает обмен данными через брандмауэр (напрямую или через прокси-сервер) к Azure, используйте Azure Monitor Application Insights. Это обеспечивает глубокий уровень отслеживания на уровне кода с поддержкой первого класса для ASP.NET, ASP.NET Core, Java, JavaScript и Node. js.

Для любого веб-приложения, доступ к которому можно получить извне, следует включить тип искусственных транзакций, известный как [наблюдение за доступностью]( https://docs.microsoft.com/azure/azure-monitor/app/monitor-web-app-availability). Важно иметь представление о том, что приложение или критическая конечная точка HTTP/HTTPS, на которой основывается ваше приложение, доступны и отвечают. Мониторинг доступности Application Insights позволяет выполнять тесты из нескольких центров обработки данных Azure и предоставлять сведения о работоспособности приложения с глобальной точки зрения.

Хотя Operations Manager способен отслеживать ресурсы, размещенные в Azure, существует несколько преимуществ, включая Azure Monitor, так как его сильные стороны пропускают ограничения в Operations Manager и устанавливают строгую основу для поддержки окончательной миграции. из него. Здесь мы рассмотрим каждый из этих рекомендаций, чтобы включить Azure Monitor в стратегию гибридного мониторинга.  

#### <a name="disadvantage-of-using-operations-manager-by-itself"></a>Недостаток использования Operations Manager

1. Анализ данных мониторинга в Operations Manager обычно выполняется с помощью предварительно определенных представлений, определенных в пакетах управления, доступных из консоли, из отчетов SQL Server Reporting Services (SSRS) или пользовательских представлений, созданных конечным пользователем. Выполнение специального анализа данных не является готовым к выполнению. Operations Manager отчеты являются негибкими, хранилище данных, которое обеспечивает долгосрочное хранение данных мониторинга, не масштабируется и не работает хорошо, а также опыт написания инструкций T-SQL, разработки Power BIного решения или использования сторонних решений. требуется для поддержки требований для различных персонажей в ИТ-организации.

2. Оповещение в Operations Manager не обеспечивает поддержку сложных выражений и включает логику корреляции, чтобы помочь уменьшить шум оповещения и сгруппировать предупреждения с целью отображения связи между ними, чтобы помочь определить основную причину Issue.

#### <a name="advantage-of-operations-manager--azure-monitor"></a>Преимущества Operations Manager и Azure Monitor

1. Azure Monitor журналы — это решение для обхода ограничений Operations Manager и дополняет базу данных хранилища данных Operations Manager для получения важных данных о производительности и журналах. Azure Monitor обеспечивает лучшую аналитику, производительность при запросе больших объемов данных и хранение данных по сравнению с хранилищем Operations Manager. Язык запросов позволяет создавать гораздо более сложные и сложные запросы с возможностью выполнять запросы через терабайты данных за считаные секунды. Вы можете быстро преобразовать данные в круговые диаграммы, диаграммы времени и многие другие визуализации. Больше не ограничивается работа с отчетами в Operations Manager на основе SQL Server Reporting Services, пользовательских запросов SQL или других обходных путей для анализа этих данных.

2. Реализация решения по управлению оповещениями Azure Monitor обеспечивает улучшенные возможности оповещений. Оповещения, созданные в группе управления Operations Manager, можно перенаправлять в рабочую область Azure Monitor Logs Analytics. Вы можете настроить подписку, ответственную за пересылку предупреждений из Operations Manager, в Azure Monitor журналы, чтобы перенаправить только определенные оповещения. Например, можно пересылать только те оповещения, которые удовлетворяют критериям для запроса в отношении поддержки управления проблемами для тенденций, а также изучения основной причины сбоев или проблем с помощью одной панели прозрачности. Кроме того, можно сопоставить другие данные журнала от Application Insights или других источников, чтобы получить представление о том, как улучшить взаимодействие с пользователем, увеличить время бесперебойной работы и сократить время на устранение инцидентов.

3. Отслеживайте облачную инфраструктуру и приложения с простой или многоуровневой архитектуры в Azure и используйте Operations Manager для мониторинга локальной инфраструктуры. Сюда входят одна или несколько виртуальных машин, несколько виртуальных машин, размещенных в группе доступности или масштабируемом наборе виртуальных машин, или контейнерное приложение, развернутое в службе Azure Kubernetes Service (AKS), работающей в контейнерах Windows Server или Linux.

4. Используйте Проверка работоспособности System Center Operations Manager решение для упреждающего оценки риска и работоспособности группы управления System Center Operations Manager с постоянным интервалом. Это может заменить или добавить пользовательские функциональные возможности, добавленные в группу управления.

5. Функция Map Azure Monitor для виртуальных машин позволяет отслеживать стандартные метрики подключения от сетевых подключений между виртуальными машинами Azure и локальными виртуальными машинами. Эти метрики включают время ответа, количество запросов в минуту, пропускную способность трафика и ссылки. Можно определить неудачные подключения, устранить неполадки, выполнить проверку миграции, выполнить анализ безопасности и проверить общую архитектуру службы. Map может автоматически обнаруживать компоненты приложений в системах Windows и Linux и сопоставлять взаимодействие между службами. Это помогает обнаруживать неосведомленные подключения и зависимости, планировать и проверять миграцию в Azure, а также сокращать согласованность во время разрешения инцидентов.

6. С помощью Монитор производительности сети Отслеживайте сетевое подключение между:

   - Ваша корпоративная сеть и Azure.

   - Критически важные многоуровневые приложения и микрослужбы.

   - Расположения пользователей и веб-приложения (HTTP/HTTPs).

   Эта стратегия обеспечивает видимость сетевого уровня без необходимости использования протокола SNMP. Она также может применяться в интерактивной карте топологии, топологии пересылки маршрутов между исходной и целевой конечными точками. Это лучший вариант, чем попытка выполнить тот же результат с наблюдением за сетью в Operations Manager или другими средствами мониторинга сети, используемыми в среде в настоящее время.

### <a name="monitor-with-azure-monitor"></a>Мониторинг с помощью Azure Monitor

Хотя миграция в облако представляет собой множество проблем, она также включает ряд возможностей. Она позволяет вашей организации переходить с одного или нескольких локальных средств мониторинга Enterprise на не только потенциально сокращать капитальные расходы и эксплуатационные расходы, но и преимущества платформы Cloud Monitoring, такой как Azure Monitor в масштабе облака. Изучите требования к мониторингу и предупреждению, конфигурацию существующих средств мониторинга, рабочие нагрузки, которые переходят в облако, и настройте Azure Monitor после завершения плана.

- Отслеживайте гибридную инфраструктуру и приложения, используя простую или многоуровневую архитектуру, в которой компоненты размещаются между Azure, другим поставщиком облачных служб и корпоративной сетью. Сюда входят одна или несколько виртуальных машин, несколько виртуальных машин, размещенных в группе доступности или масштабируемом наборе виртуальных машин, или контейнерное приложение, развернутое в службе Azure Kubernetes Service (AKS), работающей в контейнерах Windows Server или Linux.

- Включите Azure Monitor для виртуальных машин, Azure Monitor для контейнеров и Application Insights для обнаружения и диагностики проблем между инфраструктурой и приложениями. Для более тщательного анализа и корреляции данных, собранных из нескольких компонентов или зависимостей, поддерживающих приложение, необходимо использовать журналы Azure Monitor.

- Создавайте интеллектуальные оповещения, которые могут применяться к основному набору приложений и компонентов служб, чтобы уменьшить шум предупреждения с динамическими пороговыми значениями для сложных сигналов и использовать агрегирование предупреждений на основе алгоритмов машинного обучения, чтобы быстро определить Issue.

- Определите библиотеку запросов и панелей мониторинга для поддержки требований различных пользователей в ИТ-организации.

- Определите стандарты и методы для включения мониторинга в гибридных и облачных ресурсах, базовый план мониторинга для каждого ресурса, пороговые значения оповещений и т. д.  

- Настройте управление доступом на основе ролей (RBAC), чтобы предоставить пользователям и группам доступ только к тем ресурсам, которые необходимы им для работы с данными мониторинга из ресурсов, которые они несут за управление.

- Включите автоматизацию и самостоятельную службу, чтобы каждая группа была создана, включена и настроена для мониторинга и настройки предупреждений по мере необходимости.

## <a name="private-cloud-monitoring"></a>Мониторинг частного облака

Можно обеспечить целостный мониторинг Azure Stack с System Center Operations Manager. В частности, можно отслеживать рабочие нагрузки, выполняемые в клиенте, на уровне ресурсов, на виртуальных машинах и в инфраструктуре размещения Azure Stack (физические серверы и сетевые коммутаторы). Вы также можете добиться целостного мониторинга с помощью сочетания [возможностей мониторинга инфраструктуры](https://docs.microsoft.com/azure/azure-stack/azure-stack-monitor-health) , входящих в Azure Stack. Эти возможности помогают просматривать работоспособность и оповещения для Azure Stack региона и [службы Azure Monitor](https://docs.microsoft.com/azure/azure-stack/user/azure-stack-metrics-azure-data) в Azure Stack, которая предоставляет метрики и журналы инфраструктуры базового уровня для большинства служб.

Если вы уже собрались в Operations Manager, используйте пакет управления Azure Stack, чтобы отслеживать состояние доступности и работоспособности Azure Stack развертываний. Сюда входят регионы, поставщики ресурсов, обновления, запуски обновлений, единицы масштабирования, узлы единиц, роли инфраструктуры и их экземпляры (логические сущности, состоящие из аппаратных ресурсов). Для взаимодействия с Azure Stack в нем используются API-интерфейсы RESTFUL поставщика ресурсов работоспособности и обновления. Для мониторинга физических серверов и запоминающих устройств используйте пакет управления поставщика OEM (например, предоставляемый компанией Lenovo, Hewlett Packard или Dell). Operations Manager может самостоятельно отслеживать сетевые коммутаторы для сбора базовой статистики с помощью протокола SNMP. Наблюдение за рабочими нагрузками клиента возможно с помощью пакета управления Azure, выполнив два основных шага. Настройте подписку, которую требуется отслеживать, а затем добавьте мониторы для этой подписки.
